# React improvement plans

Generated by `improve-react` (full audit) at commit **d5b66a3**, folding in the
earlier `ponytail-audit` findings. React Doctor scored the frontend **100/100
with zero diagnostics** — every plan below is a "beyond the scan" senior finding,
not a rule violation. Each plan is self-contained; an executor needs no other
context.

Note: the frontend has **no lint or test runner** configured (see `CLAUDE.md`).
Verification for every plan is `npx tsc --noEmit` + `npx react-doctor@latest
--scope changed` + a manual behavior check on the affected route.

## Recommended order

Do **001 first** — it's the only user-facing defect (the primary download button
can hand an arm64 visitor a wrong-arch binary while the advertised `/dl` link does
not). 002–005 are independent, non-conflicting polish and can be done in any order
or batched into one commit.

## Dependencies

None. All five plans touch disjoint code (`platform-utils.ts` + `download-section.tsx`,
`all-downloads.tsx`, `install-commands.tsx`, `release-page.tsx`, `package.json`
respectively) and can be executed and reviewed independently.

## Explicitly not planned (verified as by-design)

- **Manual memoization** — React Compiler is enabled; the scanner and the codebase
  both rely on it. No `useMemo`/`memo` findings.
- **Go ↔ TS asset-picker duplication** — intentional cross-language parity,
  `ponytail:`-flagged in `platform-utils.ts`. Plan 001 removes the *second TS*
  copy, not this split.
- **README XSS pipeline** — `rehype-raw` → `rehype-sanitize` ordering and the
  custom schema are correct; plan 004 only hardens the URL-transform footgun.
- **`<details>`/`useSyncExternalStore`/focus-ring patterns** — correct by design.
